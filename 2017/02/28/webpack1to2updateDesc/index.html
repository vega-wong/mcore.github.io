webpack 2 升级概述和特性摘要

经过一定时间的沉淀，webpack2 也终于发布了正式版本 ^2.2, 笔者之前也是懒得的关注n个beta版本， 直到现在才来看一下新的官方文档， 在此也是对升级概述的原文用自己渣渣的英文水平作一些简单的翻译， 帮助一些同学更好的理解。

resolve.root, resolve.fallback, resolve.modulesDirectories

以上几个选项将会被替换成一个单独的选项 resolve.modules。 详细的用法请参考 resolving

resolve.extensions

这个选项已经不再强制要求传递一个空字符串了，从而被改动到了 resolve.enforceExtension。 详细的用法请参考 resolving

resolve.*

这里的改动有点多。 但不详细一一列出来， 因为大大多数都不常用。 详细的用法请参考 resolving

module.loaders 现在变成了 module.rules

旧的loader配置被一个更强大的规则系统取代了。 这个系统允许更多的相关loader配置。 考虑到更好的兼容， 旧的关键字 module.loaders 仍然是合法的。
但是新的命名 module.rules 更容易理解。

链式 loaders

跟webpack v1一样， 多个loader之间可以链式传递结果。 使用 rule.use 这个配置选项， use 可以多个loader。
在webpack v1里面， 常用的方式是多个loader用 ! 串联起来， 这个模式现在只支持在降级兼容关键字 module.loaders 里面使用。

自动的 -loader 模块名字扩展已经被移除

现在不再允许在使用loader的时候去掉 -loader 后缀了（必须写全名）

你仍然可以通过使用 resolveLoader.moduleExtensions 设置项让旧的写法（去掉 -loader ）生效， 但是官方并不推荐。

查看 #2986 能获知上述变更的详细理由。

json-loader 不再需要独立安装

当没有配置一个loader去处理JSON文件的时候， webpack 会自动尝试的使用 json-loader 加载处理。

我们决定这样做 是为了解决在webpack,node.js 和 browserify 不同的环境兼容。

Loader的配置项变成相对context中读取文件

在webpack v1中， 配置好的loader项解析文件是相对于匹配好的的文件。
在webpack v2中， 配置好的loader项解析文件是相对于context解析。

这种方式致力于解决某些情况下例如 npm link 或者一些在context意外以外的模块引用导致重复的模块问题。

module.preLoaders 和 module.postLoaders 已经被移除

UglifyJsPlugin 代码映射

插件 UglifyJsPlugin 的 sourceMap 选项现在的默认值由 true 变成 false 。
意思是说如果你想要给压缩过的代码使用代码映射或者让uglifyjs的警告给出正确的代码所在行的时候， 你需要给 UglifyJsPlugin 设置 sourceMap: true。

UglifyJsPlugin 警告

UglifyJsPlugin 的 compress.warnings 选项的默认值由 true 变成 false.
意思是如果你要查看uglifyjs的警告， 你需要设置 compress.warnings 为 true。

UglifyJsPlugin 不再支持让 Loaders 最小化文件的模式了

UglifyJsPlugin 不再支持让 Loaders 最小化文件模式。 minimize: true 这个选项只能写在指定的loader的options里面。 详情可查看loader的相关文档。

loaders的最小化模式将会在webpack 3或者更后的版本移除。

DedupePlugin 已被移除

webpack.optimize.DedupePlugin 已经不在需要了， 在你的配置代码移除掉吧。

BannerPlugin 改动。

BannerPlugin 不再支持两个参数的传递， 代替的方式是传入一个对象。

OccurrenceOrderPlugin 默认开启使用

不再需要显式的在配置文件里面声明使用该插件。

ExtractTextWebpackPlugin 改动

ExtractTextPlugin 1.0.0 在webpack v2 里面不能使用， 你需要指定安装 ExtractTextPlugin 的 v2 版本。

npm install --save-dev extract-text-webpack-plugin@beta

配置的改变主要是体现在插件的调用语法问题上面。

全动态的依赖表达式默认是失效的

现在一个只用表达式的依赖项（比如 require(expr)） 会创建一个空的上下文而不是一个带有完整目录的上下文。

当在es2015的模块化中无法工作时，请最好重构这部分的代码，如果无法进行修改这部分代码，你可以在 ContextReplacementPlugin 中来提示编译器做出正确处理。

你可能被告知这是不被允许的。 现在的CLI更加严格。

可用代替方案是， 这里有一个接口可以让你传递一些参数给配置项。 将来的工具项可能会依赖这个。

ES2015的代码分割

在webpack v1里面， 你可以通过调用 require.ensure 方法去实现在应用程序里面懒加载一些 chunks。

ES2015 Loader 特别声明了一个方法 import() 在运行时动态去加载 ES2015 的模块。 webpack 把 import() 处理成一个分割器， 讲需要用到的模块放到一个分割的chunk里面。

import()通过参数去获取模块的名字并且返回的一个Promise。

好消息： 现在加载chunk失败可以处理了， 因为返回的是一个Promise。

警告： require.ensure 可以通过第三个参数对chunk进行一些简单的命名。 但是 import API没有提供这样的能力。 如果你想保留原有的功能， 你可以继续使用 require.ensure.

动态表达式

现在是可以在 import() 里面使用部分的表达式。 它将会被当成CommonJS里面的表达式处理。 (webpack 会创建一个包含所有合法文件的上下文).

import() 会为每一个合法的模块创建分割chunk.

Loader的改变

变更的具体与loader的作者相关联。

缓存

Loaders现在默认开启缓存。 如果要关闭缓存必须声明。

复杂的选项

webpack 1 只支持通过 JSON.stringify 操作后的options提供给loaders.
webpack 2 现在支持传递一个JS对象给Loader的options。

使用复杂的选项伴随着一些限制。 你可能需要给没有options对象提供一个 ident 标识使其相对于每个loader更有参考性。

拥有一个标识的option对象表示它可以inline模式被引用， 例如：

require("some-loader??by-ident!resource")

这种写法在平常开发中用的不算多，但是有一种场景下会比较有用，就是当我们的loader需要去生成独立的代码片段时，如，我们在使用 style-loader 生成一个模块时，需要依赖前面的loader计算的结果。

所以当你想使用复杂的选项是告诉你的用户要注意使用ident标识。
